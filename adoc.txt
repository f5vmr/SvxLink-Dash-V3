SvxLink Node Builder Framework Summary
1. High-Level Concept
You are building a guided node construction system, not a config editor. The system generates
valid SvxLink configurations from a structured model.
2. System Architecture Layers
1 Web UI (Flask): user guided questions
2 Node Model: single source of truth (dictionary structure)
3 Validation Layer: ensures valid combinations
4 Renderer: builds svxlink.conf, event files, logic configs
5 Runtime: SvxLink service execution
6 Monitoring: read-only system observation
3. Example Node Model
node = {
'type': 'simplex',
'callsign': 'G7ABC',
'reflector': {
'enabled': True,
'host': 'ukwide.svxlink.net',
'port': 35300
},
'ident': {
'short': {'mode': 'cw', 'interval': 15},
'long': {'mode': 'voice', 'interval': 60}
},
'roger': {'mode': 'beep'},
'modules': ['parrot', 'echolink'],
'language': 'en_US'
}
4. Identification Rules
1 Short and long ident are configured separately
2 Modes: none, CW, voice, or combined
3 Renderer translates model into SvxLink flags
4 Repeater may enforce stricter defaults
5. Roger Tone System
1 Simplex: optional (RGR_SOUND_ALWAYS = 0 or 1)
2 Repeater: always enabled
3 Tone type handled in event logic layer
4 UI selects intent, renderer applies config

6. System Workflow
1 User answers guided questions
2 System builds node model
3 Validation ensures consistency
4 Renderer generates svxlink.conf
5 System applies configuration
6 Service is restarted safely
7 Monitoring provides read-only status
7. Implementation Checklist
1 Define node model structure
2 Build feature registry (modules, ident, roger)
3 Implement renderer functions
4 Separate Simplex vs Repeater policies
5 Add platform detection (NanoPi / Raspberry Pi)
6 Create validation rules
7 Build monitoring layer (later phase)
